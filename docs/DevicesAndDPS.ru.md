# Устройства и DPS

## 1. Откуда берутся устройства

Модуль умеет получать устройства несколькими путями:

- из облака через кнопку `Discover Devices`
- из локальной сети через кнопку `Scan LAN`
- вручную через `Add Device`

На практике лучший результат обычно дает комбинация:

1. `Discover Devices`
2. `Scan LAN`
3. при необходимости `Get local keys`

## 2. Как читать список устройств

В списке устройств важны следующие поля:

- `Name` — имя устройства в системе
- `Category` — категория Tuya
- `Status` — онлайн или офлайн
- `Connection`:
  - `LAN` — у устройства есть локальный IP
  - `Cloud` — устройство доступно через облако, но локальный IP не известен
- `Device ID` — уникальный идентификатор

## 3. Когда использовать `Discover Devices`

Используйте `Discover Devices`, если:

- вы настроили облако Tuya
- хотите получить устройства автоматически
- хотите подтянуть имя, категорию, часть метаданных и возможные локальные ключи

Если `Discover Devices` возвращает 0 устройств:

1. проверьте, привязан ли аккаунт приложения к проекту
2. проверьте `Region`
3. попробуйте заполнить `Linked App User UID`

## 4. Когда использовать `Scan LAN`

`Scan LAN` ищет устройства Tuya в локальной сети и показывает:

- имя
- `Device ID`
- IP-адрес
- определенную версию протокола

Это особенно полезно, если:

- после обнаружения из облака у устройств нет IP
- устройство переехало на другой IP
- вы настраиваете `Local only`

После сканирования используйте `Update and add devices`, чтобы:

- обновить IP и версию протокола для уже известных устройств
- добавить новые устройства, найденные только в локальной сети

## 5. Ручное добавление устройства

Используйте `Add Device`, если облако не настроено или нужно добавить устройство вручную.

Поля формы:

- `Device ID` — обязателен
- `Device Name` — понятное имя
- `Local Key` — нужен для локального управления
- `IP Address` — локальный IP устройства
- `Protocol Version` — версия протокола устройства
- `Category` — категория, помогающая при работе с DPS

Рекомендации:

- если точная категория неизвестна, временно оставьте `unknown`
- затем при первой возможности исправьте категорию на правильную

## 6. Что такое категория устройства

`Category` определяет, какие типовые DPS и их коды ожидаются у устройства.

Категория влияет на:

- удобство отображения DPS
- автоматические подписи и типы
- набор reference-DPS, если облачная спецификация временно недоступна

Если категория неверна:

- список DPS может быть неполным или неудобным
- подписи могут не совпадать с реальным назначением точки

## 7. Что такое DPS

`DPS` — это отдельные точки данных устройства.

Примеры:

- включение и выключение
- яркость
- цвет
- температура
- режим работы
- текущее измеренное значение датчика

Обычно одно устройство имеет несколько DPS.

## 8. Откуда берется список DPS в окне `Link DPS`

В окне привязки доступны два источника:

- `Cloud` — реальные функции и статус, полученные из облачной спецификации устройства
- `Reference` — встроенный справочник по типовой категории устройства

Практическое правило:

- если доступен `Cloud`, опирайтесь в первую очередь на него
- если доступен только `Reference`, проверяйте реальные значения через обновление статуса

## 9. Как привязать DPS к объекту системы

Откройте `Link DPS` у устройства и для нужной строки настройте:

- `Object` — системный объект
- `Property` — свойство, которое будет читаться или изменяться
- `Method` — дополнительный метод, если нужен сценарий вызова
- `Mode`:
  - обычный режим — разрешена запись, если DPS поддерживает запись
  - `R/O` — принудительно только чтение

После сохранения:

- изменение состояния устройства будет обновлять связанное свойство
- изменение связанного свойства в системе может отправлять команду на устройство

## 10. Когда включать `R/O`

Отмечайте `R/O`, если:

- параметр лучше только читать, а не писать обратно
- устройство ведет себя нестабильно при записи
- вы хотите избежать случайного управления из логики системы

Примеры подходящих DPS для `R/O`:

- температура датчика
- влажность
- уровень батареи
- диагностические флаги

## 11. Практический порядок привязки

Для лампы:

1. найдите `switch` / `switch_led`
2. свяжите с булевым свойством `status`
3. найдите `bright_value` / `bright_value_v2`
4. свяжите с числовым свойством яркости
5. при наличии `colour_data` свяжите с цветовым свойством

Для розетки:

1. свяжите основной `switch`
2. при наличии отдельных точек мощности или тока привяжите их как `R/O`

Для датчика:

1. свяжите измеряемые значения как `R/O`
2. не пытайтесь включать запись туда, где устройство только публикует статус

## 12. Как понять, что привязка настроена правильно

Признаки корректной настройки:

- при обновлении статуса меняется связанное свойство объекта
- при изменении свойства в системе устройство реагирует ожидаемо
- в окне `Link DPS` отображаются текущие значения

Если реакция некорректна:

- проверьте правильность `Category`
- сравните выбранный DPS с фактическим поведением устройства
- временно переведите спорные точки в `R/O`

## 13. Что делать после изменения IP или версии протокола

Если устройство сменило IP или стало нестабильно работать локально:

1. выполните `Scan LAN`
2. нажмите `Update and add devices`
3. при необходимости откройте устройство и проверьте `Protocol Version`

Подробные советы по ошибкам и локальным сбоям смотрите в `Troubleshooting.ru.md`.
