# Настройка Tuya

## 1. Что подготовить заранее

Перед настройкой убедитесь, что:

- устройства уже добавлены в `Tuya Smart` или `Smart Life`
- сервер `osysHome` имеет доступ в интернет
- для локального управления сервер находится в той же сети, что и устройства

## 2. Настройка облака Tuya

Базовый порядок настройки:

1. Создайте проект на [iot.tuya.com](https://iot.tuya.com/).
2. Выберите корректный `Data Center`.
3. Скопируйте `Access ID` и `Access Secret`.
4. Подключите минимум сервис `IoT Core`.
5. Привяжите аккаунт приложения через `Devices -> Link Tuya App Account`.

Если нужен подробный маршрут по пунктам интерфейса, используйте `GetStarted.ru.md`.

## 3. Поля настроек модуля

В окне `Settings` доступны следующие параметры:

- `Access ID` — идентификатор облачного проекта Tuya
- `Access Secret` — секрет облачного проекта Tuya
- `Region` — регион облака; должен совпадать с `Data Center` в проекте Tuya
- `Connection Mode` — режим работы модуля
- `Poll Interval` — интервал опроса статуса устройств в секундах
- `Linked App User UID` — дополнительный UID пользователя приложения для обнаружения устройств

Настройки применяются сразу, без перезапуска модуля.

## 4. Режимы подключения

### Cloud only

Используйте этот режим, если:

- локальная сеть недоступна
- вы хотите только обнаруживать устройства и управлять ими через облако
- у вас нет `Local Key`

Плюсы:

- проще стартовая настройка
- не требуется LAN-сканирование

Минусы:

- выше задержка команд
- выше расход лимитов Tuya Cloud

### Local only

Используйте этот режим, если:

- у вас уже есть `Device ID`, `Local Key` и IP-адреса устройств
- облачный проект не нужен или временно недоступен

Плюсы:

- быстрые команды
- минимальная зависимость от облака

Минусы:

- сложнее первый запуск
- труднее автоматически получать категории и реальные DPS-описания

### Both (Cloud + Local)

Это рекомендуемый режим для большинства установок.

Как он работает:

- облако помогает обнаружить устройства и получить их спецификацию
- локальная сеть используется для отправки команд и опроса статуса
- если локальный запрос не удался, модуль может использовать облако как запасной путь

## 5. Когда нужен `Linked App User UID`

Поле `Linked App User UID` полезно, если:

- облако подключается успешно
- кнопка `Discover Devices` возвращает 0 устройств
- аккаунт приложения уже привязан к проекту

Где искать UID:

- на сайте `iot.tuya.com`
- в разделе `Devices`
- у связанного аккаунта приложения

Практическое правило:

- если устройства нашлись и без UID, поле можно оставить пустым
- если устройств 0, сначала проверьте привязку аккаунта, затем попробуйте заполнить UID

## 6. Настройка локального управления

Для локального управления нужны:

- `Device ID`
- `Local Key`
- IP-адрес устройства
- корректная версия протокола: обычно `3.3`, `3.4` или `3.5`

Откуда взять данные:

- после `Discover Devices` часть данных подтягивается автоматически из облака
- IP-адрес можно получить через `Scan LAN`
- `Local Key` можно получить через `Get local keys`

## 7. Как получить `Local Key`

Есть два основных варианта:

### Через облако Tuya

Подходит, если:

- проект облака настроен правильно
- устройство видно в списке найденных устройств

### Через кнопку `Get local keys`

Этот способ использует логин и пароль от мобильного приложения `Tuya Smart` / `Smart Life`.

Нужно указать:

- `Email`
- `Password`
- `App / Vendor`
- `Region`

После запроса модуль покажет найденные устройства и их `Local Key`. Затем можно нажать `Apply keys to existing devices`.

Важно:

- учетные данные приложения не должны сохраняться как постоянный административный секрет
- если у вас включена двухфакторная защита или аккаунт OEM-бренда, результат зависит от конкретного бренда и региона

## 8. Выбор версии протокола

Вручную доступны:

- `3.1`
- `3.3`
- `3.4`
- `3.5`

Рекомендации:

- начните с версии, найденной автоматически
- если устройство вообще не работает локально, попробуйте соседние версии
- если устройство в основном работает, но иногда выдает `904`, `905` или `914`, это не всегда означает неверную версию

## 9. Настройка интервала опроса

Поле `Poll Interval` задает частоту проверки состояния устройств.

Важно понимать:

- интервал считается от начала poll-цикла, а не как "время запроса + еще полный интервал сверху"
- при нескольких устройствах модуль старается опрашивать их параллельно
- при одном устройстве параллельность не влияет на задержку, и основное время уходит в сам локальный запрос
- в режиме `Local only` изменение, сделанное в мобильном приложении, видно только на следующем локальном опросе
- если локальный запрос завершился с ошибкой и понадобился повтор, фактическая задержка может быть больше заданного интервала

Рекомендации:

- `30` секунд — хорошее значение по умолчанию
- `10-15` секунд — если нужна более быстрая реакция и устройств немного
- `60+` секунд — если устройств много или важнее снизить нагрузку

При режиме `Both` основной опрос обычно выгоднее делать по LAN.

### Как это выглядит на практике

Если у вас:

- `Local only`
- `Poll Interval = 5`
- одно устройство

то статус, измененный в мобильном приложении, обычно приходит в пределах следующего локального опроса, но:

- не мгновенно, потому что push-уведомлений из облака в этом режиме нет
- не всегда ровно через `5` секунд, потому что сам вызов `device.status()` тоже занимает время
- иногда позже, если устройство ответило не с первой попытки

### Новые диагностические сообщения в логе

Для анализа задержек используйте `Tuya.log` и ищите:

- `Poll cycle complete: devices=... elapsed=... interval=...`
- `Poll batch: devices=... mode=... elapsed=...`
- `Poll <device_id>: local status ready in ...s`
- `Poll <device_id>: cloud fallback ready in ...s`
- `Poll <device_id>: no status after ...s`
- `get_status <device_id>: success in ...s on attempt ...`
- `get_status <device_id>: finished without DPS in ...s after ... attempt(s)`

## 10. Минимальная рекомендуемая конфигурация

Для первого стабильного запуска:

- `Region` — как в Tuya Cloud
- `Connection Mode` — `Both (Cloud + Local)`
- `Poll Interval` — `30`
- выполнить `Discover Devices`
- выполнить `Scan LAN`
- при необходимости получить ключи через `Get local keys`

После этого можно переходить к привязке DPS в `DevicesAndDPS.ru.md`.
